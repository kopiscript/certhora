## Commands

```bash
npm run dev       # start dev server at localhost:3000
npm run build     # production build
npm run lint      # run ESLint
npx prisma migrate dev   # apply schema changes and regenerate client
npx prisma generate      # regenerate Prisma client after schema edits
npx prisma studio        # open Prisma GUI at localhost:5555
```

## Architecture

**Stack**: Next.js 16.2.6 (App Router), React 19, TypeScript, Tailwind CSS v4, Prisma v7 + Neon PostgreSQL (HTTP driver), NextAuth v4, Cloudflare R2 (S3-compatible storage).

**Auth flow**: `lib/auth.ts` configures NextAuth with a credentials provider (email + bcrypt password). Sessions use JWT strategy — `userType` and `id` are injected into both the JWT token and session object via callbacks. The catch-all route at `app/api/auth/[...nextAuth]/route.ts` handles all auth endpoints. `components/providers/auth-provider.tsx` wraps the app in `SessionProvider` (client component). Root `app/page.tsx` renders the marketing landing page (Header/Hero/Features/Pricing/Testimonials/Footer); `/login` and `/signup` are separate routes.

**Session/organizer dedup**: `lib/session.ts` exports `getCurrentSession()` and `getCurrentOrganizer(userId)`, both wrapped in React's `cache()`. Always use these instead of calling `getServerSession(authOptions)` or `prisma.organizer.findUnique` directly in layouts/pages — `cache()` dedupes identical calls within one request, so the protected layout and the page it wraps share a single session check and a single organizer query instead of each re-querying Neon. `getCurrentOrganizer` always includes `_count.events`.

**Database**: Prisma client is a singleton on `globalThis` (see `lib/prisma.ts`), using `PrismaNeonHttp` — every query is an HTTPS round-trip (no persistent connection), so avoid sequential/redundant queries; batch independent queries with `Promise.all`. Schema lives in `prisma/schema.prisma`. Core domain models: `User` (`UserType` enum), `Organizer` (`Tier` enum: FREE/STARTER/PRO/ENTERPRISE), `Subscription`, `Template`, `Event` (`EventStatus`), `Certificate` (`EmailStatus`), `EventFeedback`, `PaymentTransaction` (`PaymentStatus`) — plus NextAuth adapter models `Account`, `Session`, `VerificationToken`. Requires `DATABASE_URL` in `.env`.

**Never use `prisma.$transaction(...)`** — neither the interactive callback form nor the array/batch form. `PrismaNeonHttp` rejects both with `"Transactions are not supported in HTTP mode"` (verified by testing directly against the DB). This previously broke signup (`api/auth/register`) and event creation (`api/events`) silently with 500s. Instead, run writes as sequential `await` statements (with manual best-effort cleanup on failure, since there's no real rollback) or, for independent rows, `Promise.all`.

**Pricing/tiers**: `lib/tiers.ts` is the single source of truth for tier pricing/quota/feature data (`TIERS`, `TierKey`). Used by the `/pricing` page, the landing page's `PricingSection`, and the dashboard `BillingClient`. Update tier data there, not in the components.

**Billing/subscriptions**: `lib/billplz.ts` exports `createBill()` — calls the real Billplz API (`POST {BILLPLZ_URL}bills`, e.g. `https://www.billplz.com/api/v3/bills`) when `BILLPLZ_API_KEY`/`BILLPLZ_COLLECTION_ID`/`BILLPLZ_URL` are set (real credentials are configured in this environment's `.env`), otherwise falls back to a mock mode that returns a `/billing/mock-pay/[billcode]` URL (a dev-only unauthenticated page with "Simulate Success/Failure" buttons). The redirect/bill-pay URL is derived from `BILLPLZ_URL`'s origin (`{origin}/bills/{id}`), so switching between Billplz's production and sandbox hosts is just a matter of changing `BILLPLZ_URL`. Upgrades first open a checkout form (`CheckoutForm.tsx`, rendered from `BillingClient.tsx`) collecting first/last name, email, phone, country, payment method (`CARD`/`FPX`, passed to Billplz as the `method` param to restrict the hosted page to one gateway), and a required Terms/Privacy checkbox — organization is prefilled from `Organizer.orgName` and locked. Submitting goes through `POST /api/billing/checkout` (validates the payor fields, creates a `PaymentTransaction` with the payor/method columns, redirects to the bill URL) and applies **immediately** via `POST /api/billing/callback` (idempotent on already-resolved transactions, updates `Organizer.tier/certQuota/expiryDate` and upserts `Subscription`). That callback route accepts two shapes: the mock page posts `{billcode, status, refno}` directly, while a real Billplz webhook posts `{id, reference_1, paid, ..., x_signature}` — `reference_1` carries our `billcode` since Billplz has no external-reference field of its own, and the route verifies `x_signature` (HMAC-SHA256 over sorted `key+value` pairs joined by `|`, keyed by `BILLPLZ_X_SIGNATURE_KEY`) before trusting any real webhook payload. Downgrades go through `POST /api/billing/downgrade` and apply **at the end of the current paid period** instead, since no payment is involved: they write `Organizer.pendingTier`/`pendingCertQuota`/`pendingEffectiveDate` rather than changing the tier right away. `lib/billing.ts#applyPendingTierChange(organizerCd)` is what actually flips a lapsed pending downgrade — there's no cron in this project, so it's called from two mutating contexts instead: `generate-certificates/route.ts` (before the quota check) and `POST /api/billing/sync` (called once from `BillingClient` on mount so the UI reflects it promptly). Quota enforcement itself (`organizer.certQuota` vs. monthly usage) lives in `generate-certificates/route.ts` and predates this billing work.

**File storage**: `lib/r2.ts` uploads to Cloudflare R2 via `uploadToR2(key, body, contentType, cacheControl?)`. Certificate images reuse a stable key per `certId` (`certificates/{certId}.png`), so the generate-certificates route appends a `?v={timestamp}` cache-busting query param to the stored `imageUrl` on every regeneration — without it, CDN/browser caches keep serving the pre-update image after a template layout change.

**Template editor**: `components/template-editor/TemplateEditor.tsx` lets organizers drag the name/QR/custom-text elements over a 1200×840 canvas (rendered at `DISPLAY_W=720` via `SCALE`). Raw X/Y inputs are intentionally hidden from the UI — position is drag-only. Dragging snaps to canvas-center and to other elements' centers (`SNAP` px threshold) and renders red guide lines while active.

**Duplicate event**: "Duplicate" buttons on the events list and event detail page link to `/dashboard/events/new?duplicateFrom=<eventCode>`. That page fetches `GET /api/events/[eventCode]` and prefills name/dates/description/skills/template into editable state — no participants/certificates are ever copied (the new-event flow doesn't create them by default). The badge image file itself can't be carried over client-side, so only its preview/flag copy and the organizer must re-upload.

**Public cert view page**: `app/(public)/certs/[viewPage]/[certId]/page.tsx` is intentionally unauthenticated and marked `force-dynamic` (no static caching, since it increments `Certificate.viewCount` on every visit). `viewPage` is a cosmetic slug only — `certId` is the real lookup key.

**Certificate generation pipeline**: `POST /api/events/[eventCode]/generate-certificates` is the only entry point. Flow: apply any lapsed pending downgrade (`applyPendingTierChange`) → load pending (`emailStatus: "PENDING"`) certs for the event → enforce the monthly quota (`organizer.certQuota` minus certs created this month with `emailStatus != "PENDING"`) → build a template PNG buffer (either the organizer's uploaded template image, fetched and normalized through `sharp`, or a procedural SVG background from `buildProceduralTemplate` in `lib/certificate-generator.ts` when no template was uploaded) → `generateCertificateBatch` composites per-participant SVG overlays (name, cert ID, optional extra placeholders) plus a generated QR code (`qrcode` lib, pointing at the public cert view page via `buildCertUrl`) onto the template with `sharp`, 5 at a time — `generateCertificateImage` is the single-certificate unit. Name font size auto-shrinks to fit (`calcScaledFontSize`, 0.55×fontSize-per-char heuristic, floor `MIN_FONT_SIZE=14`). Outputs upload to R2 and certs flip to `QUEUED`; the event flips to `ACTIVE`. Email dispatch is a **separate, explicit step** from generation (see below) — `QUEUED` only means "generated and ready," not "sent."

**Certificate email delivery**: `lib/email.ts` wraps `nodemailer` behind `sendCertificateEmail(to, opts)`, using a singleton SMTP transport (`SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS`/`SMTP_FROM`) cached on `globalThis` like `lib/prisma.ts`. `lib/send-certificates.ts#sendQueuedCertificates(where, cap)` is the shared batch-send logic: pulls up to `cap` certs matching `where` whose `emailStatus` is `QUEUED`, `FAILED`, or `BOUNCED` (`RESENDABLE_STATUSES` — anything except `SENT`, and except `PENDING` since those have no generated image/URL yet), oldest `queuedAt` first, sends each via `buildCertUrl` (from `lib/certificate-generator.ts`) pointing at the public cert view page, and — **best-effort, no automatic retries, but always manually resendable** — flips each cert to `SENT` or `FAILED` and increments `Certificate.sendAttempts`, with a 300ms delay between sends to avoid tripping free-tier SMTP burst limits. A cert that failed once just needs the same send action triggered again (button click, or including it in a manual selection) — there's no separate "resend" endpoint or flow. Failures are logged via `console.error` in that catch block (check your dev terminal, or the Vercel deployment's Runtime Logs) since nothing persists the error message to the DB. `BOUNCED` is defined in the `EmailStatus` enum and is resendable like `FAILED`, but nothing actually sets a cert to `BOUNCED` yet (no bounce-webhook integration exists).

There are two entry points into `sendQueuedCertificates`, both capped at 20 certs per invocation, and **both require an explicit organizer action — there is no automatic/scheduled sending**:
- `POST /api/events/[eventCode]/send-emails` — auth-gated, organizer-owned-event-scoped, sends every resendable cert for that event. Triggered by the "Send Emails" button on the event detail page (`SendEmailsButton.tsx`) and by the participants page's per-event bulk send (active when an event is selected in the filter and nothing is manually checked).
- `POST /api/participants/send-emails` — auth-gated, takes `{ certIds: string[] }` and can span multiple events (ownership is enforced per-cert via `event.organizerCd`, non-resendable/non-owned ids are silently dropped). Backs the participants page's checkbox selection (`ParticipantsClient.tsx`): ticking rows — via per-row or per-page-header checkboxes — switches the button to "Send Selected (N)", which takes priority over the per-event bulk send whenever anything is checked.

There used to be a `GET /api/cron/send-emails` endpoint (Vercel Cron, every 5 min) meant to auto-drain anything left over after the 20-cap manual send — it's been **removed** for now; certificate generation and email sending must both be manually triggered by the organizer. If a batch exceeds the 20-cap, the organizer just clicks "Send" again.

**Cert ID format & duplication gotcha**: Cert IDs are `{organizerCd upper}{4-digit random}` (e.g. `GDG4819`). `lib/cert-id.ts` exports `generateUniqueCertId`/`generateUniqueCertIdBatch` for this, but **nothing calls them** — `app/api/events/[eventCode]/participants/route.ts` (participant/CSV import) reimplements the same collision-avoidance loop inline instead. If you touch cert ID generation, fix both places or consolidate onto `lib/cert-id.ts`.

**Path alias**: `@/*` maps to the project root (e.g. `@/lib/auth`, `@/components/...`).

**Required env vars**: `DATABASE_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`, `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`, `R2_PUBLIC_URL`. Optional: `BILLPLZ_API_KEY`, `BILLPLZ_COLLECTION_ID`, `BILLPLZ_URL`, `BILLPLZ_X_SIGNATURE_KEY` (billing falls back to mock mode when `BILLPLZ_API_KEY`/`BILLPLZ_COLLECTION_ID`/`BILLPLZ_URL` are unset — see above; `BILLPLZ_X_SIGNATURE_KEY` enables webhook signature verification and is otherwise skipped).

**Migrations need a direct DB connection, the app doesn't**: `npx prisma migrate dev`/`deploy` connects over raw TCP (port 5432), but the app's runtime queries go through `PrismaNeonHttp` over HTTPS (`lib/prisma.ts`). In sandboxed/agent environments that only permit HTTPS egress, port 5432 is unreachable — `prisma generate` (schema-only, no DB connection) still works, but migrations must be run from an environment with real Postgres connectivity (e.g. the user's own machine).
